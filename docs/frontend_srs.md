# Software Requirements Specification (SRS)
## 호서대 학칙/공지사항 멀티모달 RAG - 프론트엔드 명세서

문서 버전: v1.1  
작성일: 2026-03-21  
대상: 프론트엔드(React) 개발팀, 백엔드 연동 담당, QA

---

## 1. 목적 및 범위

이 문서는 호서대학교 공지사항/학칙 질의응답 서비스의 웹 프론트엔드 요구사항을 정의한다.  
프론트엔드는 사용자 질문 입력, SSE 기반 스트리밍 답변 렌더링, 세션별 대화 이력, 출처 표시, 오류 복구 UX를 제공해야 한다.

본 문서는 **웹 클라이언트(React)** 범위만 다루며, RAG 검색/생성 내부 알고리즘 자체는 백엔드 범위로 본다.

---

## 2. 시스템 개요

- 사용자는 웹 UI에서 질문을 입력한다.
- 프론트엔드는 백엔드의 스트리밍 API로 질문을 전달한다.
- 백엔드는 텍스트/비전 RAG 파이프라인을 수행하고 SSE chunk를 반환한다.
- 프론트엔드는 chunk를 순차 렌더링하여 "타이핑되는 답변" 경험을 제공한다.
- 최종 chunk 또는 완료 이벤트에서 출처 문서 정보를 표시한다.

---

## 3. 사용자 및 사용 시나리오

### 3.1 사용자 유형
- 재학생(주 사용자): 학사/장학/수강/규정 질의
- 교직원/조교(보조 사용자): 공지 확인, 규정 조회

### 3.2 핵심 시나리오
- S1: 사용자가 학칙 질문을 입력하고 실시간 답변을 받는다.
- S2: 사용자가 공지사항 질문 후 출처 문서를 클릭해 원문을 확인한다.
- S3: 동일 세션에서 후속 질문을 보내 멀티턴 대화를 이어간다.
- S4: 네트워크 오류 시 재시도 버튼으로 마지막 질문을 재요청한다.

---

## 4. 기능 요구사항 (Functional Requirements)

### FR-01. 질문 입력
- 단일 텍스트 입력창을 제공한다.
- Enter로 전송, Shift+Enter로 줄바꿈을 지원한다.
- 빈 문자열/공백만 입력은 전송하지 않는다.

### FR-02. 스트리밍 응답 렌더링
- `/chat/stream` 호출 후 SSE 데이터(`data: {...}`)를 실시간 파싱한다.
- `chunk` 텍스트를 순서대로 누적 렌더링한다.
- 스트리밍 중에는 "생성 중" 상태를 표시한다.

### FR-03. 스트리밍 종료 처리
- `[DONE]` 또는 연결 종료 시 메시지를 "완료 상태"로 전환한다.
- 완료 시점에 수신된 `sources`를 답변 카드 하단에 표시한다.

### FR-04. 대화 이력 관리
- `session_id` 기준으로 메시지 리스트를 유지한다.
- 사용자 메시지와 어시스턴트 메시지를 역할별 UI로 구분한다.
- 새로고침 시에도 최근 세션 복구가 가능하도록 로컬 저장(예: localStorage)한다.

### FR-05. 세션 시작/전환
- 세션이 없으면 새 세션을 생성하여 질의를 시작한다.
- 사용자에게 "새 대화 시작" 액션을 제공한다.
- 세션 전환 시 현재 스트리밍이 있으면 먼저 중단 확인을 요청한다.

### FR-06. 출처 표시
- 출처는 문서명/문서 ID/URL을 표시한다.
- URL이 존재하면 새 탭으로 열 수 있어야 한다.
- 출처가 없을 경우 "출처 정보 없음" 상태를 명확히 표시한다.

### FR-07. 에러 처리
- API 키/권한 오류, 타임아웃, 네트워크 단절, 서버 5xx를 구분하여 안내한다.
- 실패한 요청에 대해 "다시 시도" 액션을 제공한다.
- 스트리밍 중 오류가 발생하면 현재까지 받은 텍스트는 유지한다.

### FR-08. 상태 표시
- 초기 로딩, 전송 중, 스트리밍 중, 완료, 실패 상태를 시각적으로 구분한다.
- 사용자가 인지 가능한 피드백(스피너/토스트/인라인 메시지)을 제공한다.

### FR-09. 입력 제한 및 보호
- 질문 길이 제한(권장: 1~2000자)을 둔다.
- 과도한 연속 전송(예: 더블클릭) 방지 디바운스/버튼 비활성화를 적용한다.

### FR-10. 접근성
- 키보드만으로 질문 입력/전송/재시도가 가능해야 한다.
- 버튼/입력창/링크에 접근성 라벨(aria-label)을 제공한다.

## 5. 비기능 요구사항 (Non-Functional Requirements)

### NFR-01. 성능
- 첫 토큰 체감 시간(TTFT) 목표: 평균 3초 이내(내부망 기준).
- 일반 답변의 완성까지 사용자 대기 시간 목표: 15초 이내(질문 난이도에 따라 변동 가능).

### NFR-02. 안정성
- SSE 연결 중 일시적 네트워크 끊김 발생 시 사용자에게 재시도 경로를 제공한다.
- 브라우저 탭 전환/리렌더 시에도 진행 중 스트림 상태가 깨지지 않도록 한다.

### NFR-03. 사용성
- 모바일/데스크톱 반응형 레이아웃을 지원한다.
- 최소 2가지 테마(라이트/다크) 대응을 권장한다.

### NFR-04. 보안
- 브라우저에 민감 키를 저장하지 않는다.
- 내부 API 호출 시 프론트는 인증 토큰만 전달하고, 서버 간 비밀키는 노출하지 않는다.
- 로그에 개인정보/민감 질문 전문을 평문으로 남기지 않도록 마스킹 정책을 적용한다.

### NFR-05. 호환성
- 최신 Chrome/Edge 기준 정상 동작해야 한다.
- Safari/Firefox는 핵심 기능(입력, 스트리밍, 출처 열기) 호환을 보장한다.

## 6. 외부 인터페이스 요구사항

본 장의 API 내용은 `docs/api_spec.md`의 최신 표준과 동일한 기준으로 유지한다.

## 6.1 API 연동 (요약)
- Base URL: `http://<AI_ENGINE_HOST>:8000/api/v1`
- Content-Type: `application/json`
- Endpoint: `POST /chat/stream`

요청 예시:
```json
{
  "user_id": "student_123",
  "session_id": "session_abc998",
  "question": "이번 2026학년도 장학금 신청 기한이 언제야?"
}
```

SSE 응답 예시:
```text
data: {"chunk":"2026학년도 "}
data: {"chunk":"1학기 장학금 신청 기한은 ..."}
data: {"chunk":"", "sources":[{"doc_id":"notice_123","title":"2026 장학금 안내","file_url":"https://..."}]}
data: [DONE]
```

## 6.2 프론트 데이터 모델 (권장)
```json
{
  "id": "msg_uuid",
  "role": "user|assistant",
  "content": "string",
  "status": "pending|streaming|done|error",
  "sources": [
    {"doc_id":"string","title":"string","file_url":"string"}
  ],
  "created_at": "ISO-8601"
}
```

## 7. 화면 요구사항 (UI Requirements)

### UI-01. 채팅 메인 화면
- 상단: 서비스 타이틀, 세션 정보(선택)
- 중앙: 대화 버블 리스트(사용자/AI 구분)
- 하단: 질문 입력창 + 전송 버튼 + 재시도 버튼(오류 시)

### UI-02. 답변 카드
- 본문: 마크다운 렌더링(볼드, 리스트, 링크)
- 하단: 출처 목록(문서명 + 링크)
- 상태 배지: 생성중/완료/오류

### UI-03. 빈 상태
- 첫 진입 시 예시 질문 3개(장학금, 수강신청, 학칙 조항)를 제시한다.

---

## 8. 예외/오류 시나리오

- E1: SSE 연결 실패 -> 인라인 오류 + "다시 시도"
- E2: 응답 중간 중단 -> 수신 텍스트 유지 + "이어서 재질문" 유도
- E3: 출처 누락 -> 답변은 표시하되 출처 섹션에 누락 안내
- E4: 잘못된 요청 본문(4xx) -> 사용자 입력 검증 메시지 노출

---

## 9. 수용 기준 (Acceptance Criteria)

- AC-01: 질문 전송 후 SSE chunk가 실시간으로 누적 렌더링된다.
- AC-02: 완료 시점에 출처 정보가 화면에 표시된다.
- AC-03: 네트워크 오류 발생 시 사용자가 재시도할 수 있다.
- AC-04: 세션별 대화 이력이 유지되고 새 대화 시작이 가능하다.
- AC-05: 데스크톱/모바일에서 입력, 스트리밍, 출처 링크가 정상 동작한다.

---

## 10. 구현 권장사항 (Frontend)

- 상태 관리: React Query + Zustand(또는 Context) 조합 권장
- SSE 처리: `EventSource` 또는 `fetch` 스트림 파서 일관화
- 메시지 렌더링: 낙관적 UI(사용자 질문 즉시 표시) 적용
- 마크다운: `remark` 계열 사용 시 링크 보안 옵션(`rel="noreferrer"`) 적용

---

## 11. 오픈 이슈 (백엔드 협의 필요)

- OI-01: `/chat/stream`의 오류 이벤트 포맷 표준화 필요
- OI-02: `sources` 스키마 필드 확정(`doc_id`, `title`, `file_url`, `page` 등)
- OI-03: 세션 히스토리 조회/삭제 API의 최종 엔드포인트 확정 필요
- OI-04: 텍스트 RAG/비전 RAG 라우팅 결과를 프론트에 노출할지 여부 결정 필요
